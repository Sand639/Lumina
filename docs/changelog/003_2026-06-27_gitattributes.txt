ブランチ : feature/Initialize

--------------------------------------------------------------------------------
■ このコミットで何をしたか（概要）
--------------------------------------------------------------------------------
リポジトリ直下に .gitattributes を追加し、改行コードの扱いとファイル種別
(テキスト/バイナリ)を Git に明示した。これでコミット時に毎回出ていた
「LF will be replaced by CRLF」警告を抑え、改行をリポジトリ全体で統一する。

--------------------------------------------------------------------------------
■ 変更したファイル
--------------------------------------------------------------------------------
[新規] .gitattributes

--------------------------------------------------------------------------------
■ なぜそうしたか
--------------------------------------------------------------------------------
・改行コードをリポジトリ内で「LF」に統一するため。
   Windows のエディタは CRLF、Git の標準は LF。混在するとファイル全体が
   「変更扱い」になって差分が汚れたり、別環境との差で揉める。
   .gitattributes で方針を固定し、誰の環境でも同じ結果になるようにする。

・VS のプロジェクトファイルだけは CRLF に固定するため。
   Visual Studio は .sln/.vcxproj 等を CRLF 前提で読み書きするので、
   ここだけ eol=crlf を明示して VS との相性問題を避ける。

・バイナリを「変換しない」と宣言するため。
   画像/モデル/.lib/.exe 等に改行変換や差分計算が走るとファイルが壊れる/重くなる。
   binary 指定でそれを止める。

--------------------------------------------------------------------------------
■ 使っている技術・用語の解説
--------------------------------------------------------------------------------
・.gitattributes
   パスごとに Git の扱いを指定する設定ファイル。改行(eol)、テキスト/バイナリ判定、
   差分方法などをルール化できる。リポジトリに含めるので全員に共有される。

・text=auto
   テキストと判定したファイルを、リポジトリ内では LF に正規化して保存する。
   取得(チェックアウト)時は環境に合わせて展開される(Windowsなら CRLF)。

・eol=crlf / eol=lf
   そのファイルの改行を固定する。VS プロジェクト系は crlf に固定した。

・binary
   「-text -diff」と同じ意味。改行変換も差分計算もしない=ファイルをそのまま保持。

・git add --renormalize .
   既存ファイルへ新しい .gitattributes ルールを適用し直すコマンド。
   今回はリポジトリ内が既に LF だったため、実変更は無し(=クリーン)だった。

--------------------------------------------------------------------------------
■ 結果
--------------------------------------------------------------------------------
.gitattributes のみ追加。既存ソースの再正規化による差分は発生せず。
以後、改行起因の警告とノイズ差分が出にくくなる。
